Задача: довести скачивание в Safari до нужного поведения, не затрагивая другие функции.

1. Что уже понятно по коду

- В `src/components/dashboard/History.tsx` логика истории уже разделяет мобильное и десктопное поведение:
  - на мобильных одиночное изображение открывается через `window.open(...)`
  - на десктопе файл собирается в `blob` и сохраняется
- Но сохранение идет через `safeBlobDownload` из `src/lib/telegram.ts`.
- Сейчас `safeBlobDownload` обрабатывает весь Safari одинаково: вместо прямого скачивания открывает `blob:`-ссылку в новой вкладке.
- Из-за этого desktop Safari ведет себя не как обычный ПК-браузер: ZIP/файл может открываться, а не скачиваться напрямую.

2. План исправления

- Менять только историю загрузок, без затрагивания остальных сценариев приложения.
- В `src/components/dashboard/History.tsx` добавить локальное различение:
  - iPhone/iPad/iPod
  - iPadOS в desktop mode (`MacIntel + maxTouchPoints > 1`)
  - desktop Safari как отдельный случай
- В этом же файле сделать локальный helper для скачивания `blob`:
  - iOS/iPadOS Safari: сохранить текущее поведение для одиночной картинки — открытие в новой вкладке
  - desktop Safari и остальные десктопы: принудительное скачивание через `a[download] + URL.createObjectURL(...)`
  - освобождение `ObjectURL` делать с задержкой, чтобы не ловить Safari race-condition
- Перевести на этот helper только скачивания внутри `History.tsx`:
  - одиночная картинка
  - ZIP-архив кластера
  - при необходимости остальные blob-скачивания на этой странице, чтобы поведение в Safari было единообразным

3. Что останется без изменений

- Telegram-логика
- логика выбора нужного кластера
- логика сборки ZIP по выбранным изображениям
- другие страницы/компоненты
- общая утилита `safeBlobDownload`, чтобы не задеть другие функции приложения

4. Ожидаемый результат

- Mobile Safari: одиночное изображение открывается в новой вкладке
- Desktop Safari: одиночное изображение скачивается сразу файлом
- Desktop Safari: архив формируется как ZIP и скачивается файлом, а не открывается как blob-вкладка
- В остальных браузерах текущее рабочее поведение не меняется

5. Технические детали

- Причина бага не в самой сборке ZIP, а в финальном способе сохранения `blob` для Safari.
- Полагаться только на `useIsMobile()` нельзя: для Safari важно различать именно iOS/iPadOS и desktop Safari, а не только ширину экрана.
- Для iPadOS нужен отдельный детект, потому что он может маскироваться под desktop Safari.

6. Проверка после внедрения

- macOS Safari: скачать одно изображение из истории
- macOS Safari: скачать ZIP для генерации с несколькими карточками
- iPhone/iPad Safari: проверить, что одиночная картинка открывается в новой вкладке
- проверить, что ZIP берется именно из нажатого кластера, а не из первого попавшегося  
  
Все остальные функции сервиса оставить без изменений и не затрагивать, только те, что нужно для реализации этой задачи.
