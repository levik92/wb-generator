

# Диагностика: ошибка регенерации до отправки в edge-функцию

## Что подтвердил скринкаст
Пользователь нажимает «Перегенерировать» → сразу появляется тост «Ошибка перегенерации». В БД нет ни списания, ни job — значит `supabase.functions.invoke(...)` НЕ вызывается. Код падает в одном из ранних `throw` внутри `regenerateCard` (строки 1229–1402 в `src/components/dashboard/GenerateCards.tsx`), и общий catch-блок (строка 1388) скрывает реальную причину под одинаковым текстом.

## Где конкретно может падать (в порядке вероятности)

1. **`throw new Error('Оригинальные изображения недоступны')`** (строка 1326) — `allProductImages.length === 0`. У этого пользователя 35 успешных регенераций в истории, но он мог зайти на старую генерацию, у которой:
   - `image.jobId` отсутствует (карточка из старой схемы),
   - `currentJobId` уже сброшен,
   - fallback по `generation_tasks.job_id` не нашёл запись (taskId — это id из `generations.output_data`, а не реальный `generation_tasks.id`),
   - `jobData.productImages` и `uploadedProductImages` пусты, потому что страница перезагружалась.
2. **`throw new Error('Название товара недоступно')`** (строка 1338) — `productNameToUse` пустой по той же причине.
3. **Ранний return по `!activeModel`** (строка 1234) показывает другой тост («Подождите»), но если хук успел вернуть пустой объект — может пройти дальше и упасть в `getImageEdgeFunctionName`.

Подтверждение из БД: за сегодня у пользователя ноль job'ов с `category='regeneration'` и ноль списаний по `-1` токену → клиент действительно не доходит до invoke.

## Что нужно сделать

### 1. Различить ошибки в catch и показать пользователю причину
Файл `src/components/dashboard/GenerateCards.tsx`, функция `regenerateCard` (строки 1388–1402).

Сейчас любая ошибка показывает один и тот же текст. Заменить на маппинг:
- `error.message === 'Оригинальные изображения недоступны'` → тост «Не удалось найти исходные изображения. Откройте генерацию заново и попробуйте снова».
- `error.message === 'Название товара недоступно'` → тост «Не удалось определить название товара. Перезагрузите страницу».
- сетевые ошибки (`FunctionsFetchError`, `error.context?.status`) → «Не удалось связаться с сервером. Проверьте интернет».
- остальное → текущий общий текст.

### 2. Добавить расширенный лог перед каждым throw
Перед строками 1326 и 1338 добавить `console.error('[Regenerate] aborted', { reason, image, jobIdForRegeneration, hasJobData: !!jobData, uploadedCount: uploadedProductImages.length })`. Это даст конкретные данные в следующем скринкасте.

### 3. Усилить fallback по `taskId` в БД
В блоке 1264–1274 текущий fallback ищет `generation_tasks.id = image.id`. Но `image.id` в History — это сгенерированный uuid из `generations.output_data.images[].id`, а не id строки в `generation_tasks`. Добавить:
- если `image.id` не найден в `generation_tasks`, поискать по `generations` где `id = image.generationId` или `output_data->>job_id` содержит соответствующий job;
- если `image` пришёл из History — у него должно быть поле с `generation_id` или `job_id`, использовать его напрямую.

### 4. Сбросить «зависшую» кнопку при размонтировании
Добавить `useEffect(() => () => setRegeneratingCards(new Set()), [])`, чтобы возврат на вкладку не оставлял задизейбленные кнопки.

### 5. (Опционально) Логировать клиентские ошибки на сервер
Через существующий `useSecurityLogger.logSuspiciousActivity` отправлять `{ activityType: 'regeneration_client_error', metadata: { reason, image_id, has_job_id } }`. Это позволит без скринкастов видеть реальные причины у других пользователей.

## Что попросить у пользователя дополнительно
После выкатки фикса — повторить попытку и прислать новый скриншот тоста. Если это вариант №1, то к нему добавится пояснение «откройте генерацию заново», и проблема будет понятна без обращения в поддержку.
